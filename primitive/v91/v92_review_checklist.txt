A2. esde_v90_calibrate.py: sub-window torque injection
------------------------------------------------------------
[ ] --feedback-interval 引数追加（default=50）
[ ] run()にfeedback_interval引数追加

[ ] window loop内のロジック:
    interval >= 50: 従来通りstep_window 1回
    interval < 50:
      - n_chunks = window_steps // interval（50/10=5 chunks等）
      - 中間chunk: virtual_enabled=False でstep_window(interval)
        → window_count -=1（premature increment修正）
        → apply_torque_only()
        → virtual_enabled復元
      - 最終chunk: full step_window(interval)

REVIEWER:
核心的判断: 中間chunkでvirtual_enabled=Falseにする。
これでvirtual.step()（birth/share/cull）がスキップされる。
最終chunkでvirtual_enabled=Trueに戻してfull処理。

window_count -= 1 の理由:
step_window()は末尾でwindow_count += 1する。
中間chunkでも呼ばれるため、n_chunks回incrementされてしまう。
最終chunk分の1回だけ有効にするため、中間chunkでdecrementする。

REVIEWER WARNING:
step_window()内のstress equilibriumはchunk毎に走る。
50 step分のstressと10 step分のstressは異なる可能性。
ただしstressはalive_linksのEMAなので、
短いchunkでも大きな差は出ないはず。

------------------------------------------------------------

------------------------------------------------------------

[ ] detail.json metaに"feedback_interval"追加

SECTION D: sub-window injection の正確性
================================================================

[ ] interval=50の場合:
    n_chunks = 50/50 = 1。
    ループは1回。chunk=0 = n_chunks-1なので最終chunk。
    full step_window(50)。従来と完全同一動作。

[ ] interval=10の場合:
    n_chunks = 50/10 = 5。
    chunk 0-3: virtual_enabled=False, step_window(10), torque_only
    chunk 4: full step_window(10)
    合計: 50 physics steps + 4回追加torque + 1回full virtual
    
    REVIEWER: 従来は50 step後に1回torque。
    interval=10では10 step後にtorque、計5回。
    Mは前windowの値を使い続ける（window内で更新されない）。
    
    これは「50 stepのθ drift」を「10 stepの小さなdrift × 5回補正」に分解。
    Takaの仮説を直接テスト。

[ ] interval=5の場合:
    n_chunks = 50/5 = 10。
    chunk 0-8: virtual_enabled=False, step_window(5), torque_only
    chunk 9: full step_window(5)
    合計: 50 physics steps + 9回追加torque + 1回full virtual

[ ] window_countの正確性:
    中間chunk: step_window() → window_count+1 → 直後に-1
    最終chunk: step_window() → window_count+1（これが有効な1回）
    window_countの進行はintervalに関わらず1 window/ループ。

SECTION E: 性能影響
================================================================

interval=50: 従来通り。14-17秒/window。

interval=10:
  step_window × 5回。physicsコストは同じ（合計50 steps）。
  ただしstep_window()のオーバーヘッド（island検出、stress等）が
  5回走る。island検出が支配的（≈0.5秒/回）。
  推定: 14-17秒 → 16-20秒/window。約20%増。

interval=5:
  step_window × 10回。
  推定: 14-17秒 → 19-25秒/window。約40-50%増。

3-seed sanityでは問題なし。48-seedは要判断。

REVIEWER WARNING:
step_window()はisland検出を毎回行う。
中間chunkではisland検出は不要（virtual.stepが走らないため）。
しかしstep_window()の中を変えずに呼んでいるので回避できない。
性能が問題になるなら、engine側の修正が必要。
今は3-seed sanityなので保留。
